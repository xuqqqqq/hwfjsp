# 华为铝锭加工复杂约束柔性作业车间调度问题(FJSP)需求文档

## 1. 项目概述

### 1.1 项目背景
本项目旨在开发一个针对铝锭加工行业的复杂约束柔性作业车间调度求解程序。调度场景以铝锭加工为例，涉及冷轧、退火、精整等多道工序，需要在满足多种复杂约束条件下，优化生产调度方案。

### 1.2 问题规模
- **订单数量**：约1500个铝卷订单，其中1000+在制卷
- **设备数量**：约50台机器
- **机器类型**：冷轧机、氮保炉、拉弯矫、纵剪、高速切边机、纯拉伸机组、厚板剪、清洗线、铝箔轧机、重卷机、汽车板异形剪、汽车板连退线、汽车板表面处理线

### 1.3 典型加工流程
铝锭经过热轧后的加工流程：
1. **开胚冷轧工序**：首次冷轧
2. **中间冷轧工序（可重入）**：
   - 送至精整线进行清洗
   - 送入退火炉进行中间退火
   - 送回冷轧线继续冷轧
   - 该工序可以循环数次
3. **成品冷轧工序**：最终冷轧
4. **精整工序**：成品精整
5. **纵剪工序**：形成成品铝卷

## 2. 核心数据结构与概念

### 2.1 工厂(Factory)
```cpp
struct Factory {
    Id_t id;           // 工厂唯一标识符
    std::string name;  // 工厂名称
};
```

**说明**：系统支持多工厂调度，工厂之间可能涉及跨厂转运。

### 2.2 设备(Equipment)
```cpp
struct Equipment {
    Id_t id;                                              // 设备全局唯一ID
    std::string name;                                     // 设备名称
    bool batch_type;                                      // 是否为组批设备
    Id_t factory_id;                                      // 所属工厂ID
    std::vector<std::pair<Time_t, Time_t>> down_interval; // 设备不可用时间区间
};
```

**关键属性**：
- **batch_type**: 标识是否为p-batch组批机台
- **down_interval**: 设备维修时段，在此期间设备不可用

### 2.3 任务/订单(Task)
```cpp
struct Task {
    Id_t id;                                // 任务全局唯一ID
    std::string name;                       // 任务名称
    Time_t earliest_start_time;             // 最早开始时间（释放时间）
    Time_t delivery_deadline;               // 交付截止时间
    double priority;                        // 任务优先级权重
    Weight_t final_output_weight;           // 最终产出重量
    std::vector<ProcessPath> process_paths; // 可选加工路径集合
};
```

**关键属性**：
- **earliest_start_time**: 工件到达时间，工件在此时间前不能开始加工
- **priority**: 优先级权重，数值越小优先级越高
- **process_paths**: 支持多条替代加工路径可选

### 2.4 加工路径(ProcessPath)
```cpp
struct ProcessPath {
    std::vector<Process> processes;  // 路径中的工序序列
    QTimeManager queue_time;         // 路径的时间约束管理器
};
```

**说明**：同一任务可能有多条不同的加工路径，求解器需要选择最优路径。

### 2.5 工序(Process)
```cpp
struct Process {
    Id_t id;                                             // 工序全局唯一ID
    int identifier;                                      // 在路径中的序号标识
    bool batch_type;                                     // 是否为组批工序
    std::vector<std::pair<Id_t, Id_t>> diff_factory_info;// 允许的跨厂转运配置
    std::vector<std::string> batch_family;               // 组批属性信息
    std::set<ProcessEquipment> candidate_equipments;     // 候选加工设备集合
};
```

**关键属性**：
- **candidate_equipments**: 支持FJSP特性，每道工序可在多台机器上加工
- **batch_family**: 组批家族信息，用于判断工序能否组批。两个工序能否组批需要对各自的 `batch_family` 中的字符串进行两两模糊匹配，只要存在任意一对字符串能够模糊匹配，则这两个工序可以组批
- **diff_factory_info**: 定义允许的跨厂转运路径

**组批匹配规则**：
```
判断工序A和工序B能否组批:
  for str_a in batch_family[A]:
    for str_b in batch_family[B]:
      if fuzzy_match(str_a, str_b):  // 模糊匹配
        return True  // 可以组批
  return False  // 不可组批
```

### 2.6 工序设备加工信息(ProcessEquipment)
```cpp
struct ProcessEquipment {
    Id_t equipment_id;    // 加工设备ID
    double priority;      // 机器优先级权重（数值越小优先级越高）
    Time_t process_time;  // 加工时间
    int batch_size;       // 占用组批机器的空间比例（1/batch_size）
};
```

**说明**：同一工序在不同设备上的加工时间可能不同。

### 2.7 时间间隔约束(QTime)
```cpp
struct QTime {
    Id_t start_id;                      // 前序工序ID
    ProcessType start_type;             // 前序工序时间点类型(开始/结束)
    Id_t end_id;                        // 后续工序ID
    ProcessType end_type;               // 后续工序时间点类型(开始/结束)
    std::optional<Time_t> min_interval; // 最小时间间隔约束
    std::optional<Time_t> max_interval; // 最大时间间隔约束
};

enum class ProcessType {
    StartOfProcess,  // 工序开始时间点
    EndOfProcess,    // 工序结束时间点
    Any,             // 任意时间点
};
```

**说明**：
- **min_interval**: 紧邻工序之间的最小生产间隔
- **max_interval**: 同一工件的前后工序（不一定紧邻）之间的最大生产间隔

## 3. 十二大约束特性详解

### 3.1 特性1：柔性作业车间调度(FJSP)

**定义**：每道工序可以在多台机器上加工，不同机器的加工时间可能不同。

**实现方式**：
- `Process::candidate_equipments` 存储所有候选设备及其加工时间
- 求解器需要为每个工序选择一台设备进行加工

**示例**：
```
工序A可以在以下设备加工：
- 设备1：加工时间 120 分钟
- 设备2：加工时间 150 分钟
- 设备3：加工时间 100 分钟
```

---

### 3.2 特性2：释放时间(Release Time)

**定义**：每个工件和机器都有最早可用时间，在此时间之前不能开始加工。

**实现方式**：
- **工件释放时间**: `Task::earliest_start_time`
- **机器释放时间**: 通过 `Equipment::down_interval` 的反向推导，或在初始状态中设置

**约束**：
- 工件的第一道工序开始时间 ≥ `earliest_start_time`
- 机器在其最早可用时间之前不能安排任何工序

**示例**：
```
订单A: earliest_start_time = 480 (8小时后到达)
设备M1: 当前时间为0，但设备正在加工其他订单，最早可用时间为300
```

---

### 3.3 特性3：最大生产间隔(Maximum Time Lag)

**定义**：同一工件的前后工序（不一定紧邻）之间的相对开工时间有最大间隔限制。

**实现方式**：`QTime::max_interval`

**约束公式**：
```
如果 QTime(op_i, op_j) 定义了 max_interval:
  T_start(op_j) - T_ref(op_i) ≤ max_interval
  
其中 T_ref 可以是:
  - T_start(op_i): op_i 的开始时间 (start_type = StartOfProcess)
  - T_end(op_i): op_i 的结束时间 (start_type = EndOfProcess)
```

**应用场景**：
- 中间冷轧后需要在规定时间内完成退火，防止材料性能退化
- 精整后需要在一定时间内完成纵剪，防止表面氧化

**示例**：
```
工序A(清洗) -> 工序B(退火)
max_interval = 180 分钟
约束: T_start(B) - T_end(A) ≤ 180
```

---

### 3.4 特性4：最小生产间隔(Minimum Time Lag)

**定义**：同一工件的紧邻工序之间的相对开工时间有最小间隔限制。

**实现方式**：`QTime::min_interval`

**约束公式**：
```
如果 QTime(op_i, op_{i+1}) 定义了 min_interval:
  T_start(op_{i+1}) - T_end(op_i) ≥ min_interval
```

**应用场景**：
- 冷轧后需要自然冷却一段时间才能进入下一道工序
- 清洗后需要晾干时间
- 退火后需要降温时间

**示例**：
```
工序A(冷轧) -> 工序B(清洗)
min_interval = 30 分钟
约束: T_start(B) - T_end(A) ≥ 30 (冷却时间)
```

---

### 3.5 特性5：顺序相关切换时间(Sequence-Dependent Setup)

**定义**：工序与工序之间在同一设备上切换需要时间，切换时间取决于前后工序的类型，切换矩阵非对称。

**实现方式**：`SetupTimeTable` - 非对称矩阵

**约束**：
```
如果工序i和工序j在同一设备m上先后加工:
  T_start(j, m) ≥ T_end(i, m) + SetupTime(i, j, m)
```

**特点**：
- **非对称性**: SetupTime(i, j) ≠ SetupTime(j, i)
- **序列相关**: 切换时间取决于前后工序的具体类型

**示例**：
```
设备M1上的切换时间矩阵:
        → 工序类型A  工序类型B  工序类型C
工序类型A     0         20         30
工序类型B     15        0          25
工序类型C     40        35         0

解释: 从类型A切换到类型B需要20分钟，但从类型B切换到类型A只需15分钟
```

---

### 3.6 特性6：可重入(Re-entrant)

**定义**：某些工序（如中间冷轧）可以重复执行多次，不一定只执行一次。

**实现方式**：
- 在 `ProcessPath::processes` 中，同类型工序可以出现多次
- 通过工艺路径设计支持循环加工

**典型流程**：
```
开胚冷轧 → [精整 → 退火 → 冷轧] × N次 → 成品冷轧 → 精整 → 纵剪
         ↑__________________________|
              (可重入循环)
```

**约束**：
- 循环次数可能因订单而异
- 每次重入都要满足相应的时间约束和设备约束

**示例**：
```
任务A的加工路径:
  op1: 开胚冷轧
  op2: 精整
  op3: 退火
  op4: 中间冷轧 (第1次)
  op5: 精整
  op6: 退火
  op7: 中间冷轧 (第2次，重入)
  op8: 成品冷轧
  op9: 纵剪
```

---

### 3.7 特性7：替代加工路径可选(Alternative Process Routes)

**定义**：同一任务可能有多条不同的加工路径可选，求解器需要选定加工路径。

**实现方式**：`Task::process_paths` - 存储多条路径

**约束**：
- 每个任务只能选择其中一条路径执行
- 不同路径可能包含不同的工序、不同的设备选择

**决策变量**：
```
route_selection[task_id] ∈ {0, 1, ..., num_paths-1}
```

**示例**：
```
任务A有两条可选路径:

路径1: 开胚冷轧 → 精整 → 退火 → 冷轧 → 纵剪
       (快速路径，质量标准)

路径2: 开胚冷轧 → 精整 → 退火 → 冷轧 → 精整 → 退火 → 冷轧 → 纵剪
       (高质量路径，多次循环)
```

---

### 3.8 特性8：考虑设备维修时段(Maintenance Windows)

**定义**：设备在给定的维修时段内不可用，不能安排任何工序。

**实现方式**：`Equipment::down_interval` - 存储不可用时间区间

**约束**：
```
对于设备m的维修时段 [mt_start, mt_end]:
  如果工序op在设备m上加工，则:
    T_end(op) ≤ mt_start  OR  T_start(op) ≥ mt_end
```

**特点**：
- 支持多个维修时段
- 维修时段为硬约束，不可违反

**示例**：
```
设备M1的维修时段:
  [1440, 1500] (第1天的24:00-25:00，即次日0:00-1:00)
  [2880, 2940] (第2天的24:00-25:00)

约束: 任何在M1上的工序不能跨越这些时段
```

---

### 3.9 特性9：考虑跨厂转运(Inter-Factory Transportation)

**定义**：加工过程中涉及跨厂转运，从一个工厂的设备加工后转运到另一个工厂的设备继续加工。

**实现方式**：
- `TransitionTimeTable`: 存储工厂间转运时间矩阵
- `Process::diff_factory_info`: 定义允许的跨厂转运路径

**约束**：
```
如果工序op_i在工厂f1完成，工序op_{i+1}在工厂f2开始:
  T_start(op_{i+1}) ≥ T_end(op_i) + TransitionTime(f1, f2)
  
前提: (f1, f2) ∈ Process::diff_factory_info[op_{i+1}]
```

**特点**：
- 转运时间可能较长
- 不是所有工序都允许跨厂加工
- 需要考虑转运成本和时间

**示例**：
```
转运时间矩阵:
        → 工厂A  工厂B  工厂C
工厂A     0      120    180
工厂B     120    0      90
工厂C     180    90     0

工序op5允许的跨厂转运:
  diff_factory_info = [(A, A), (A, B), (B, B)]
  不允许转运到工厂C
```

---

### 3.10 特性10：考虑工件优先级(Job Priority)

**定义**：不同工件有不同优先级，需要优先加工优先级高的工件。

**实现方式**：`Task::priority` - 优先级权重

**优化目标**：
```
最小化加权总完工时间:
  Minimize: Σ (priority[i] × completion_time[i])

或加权总延迟:
  Minimize: Σ (priority[i] × max(0, completion_time[i] - deadline[i]))
```

**约束与策略**：
- 优先级高的任务应尽早完成
- 在资源冲突时，优先分配给高优先级任务
- 数值越小，优先级越高

**示例**：
```
订单A: priority = 1.0  (紧急订单)
订单B: priority = 2.0  (普通订单)
订单C: priority = 0.5  (特急订单，最高优先级)

调度时优先安排: C → A → B
```

---

### 3.11 特性11：并行组批加工(Parallel Batch Processing)

**定义**：部分机台为p-batch机台，可以同时加工多个工件，但只有属于同一family的任务才能放在一起组批，批次加工时长取批次内工件时长的最大值。

**实现方式**：
- `Equipment::batch_type = true`: 标识组批设备
- `Process::batch_family`: 定义工序的家族属性
- `ProcessEquipment::batch_size`: 工序占用批次容量的比例

**关键规则**：

1. **家族兼容性（模糊匹配）**：
   ```
   两个工序op_i和op_j能否组批的判断:
     for str_i in batch_family[op_i]:
       for str_j in batch_family[op_j]:
         if fuzzy_match(str_i, str_j):  // 模糊匹配
           return True  // 可以组批
     return False  // 不可组批
   
   说明：
   - 不需要精确字符串匹配
   - 字符串数量要一致
   - 只要batch_family中的一对字符串能够模糊匹配即可
   - 例如 A* 和 A1 能够匹配，*a1 和 aa1 能够匹配 ……
   ```

2. **容量约束**：
   ```
   Σ (1 / batch_size[op_i]) ≤ 1  (对于同一批次)
   ```

3. **时间计算**：
   ```
   批次开始时间: T_start(batch) = max(T_ready[op_i] for op_i in batch)
   批次结束时间: T_end(batch) = T_start(batch) + max(process_time[op_i] for op_i in batch)
   ```

**约束**：
- 同一批次内的所有工序同时开始、同时结束
- 批次容量不能超过1（100%）
- 只有batch_family能够模糊匹配的工序才能组批

**示例**：
```
设备M1是组批设备:
  batch_type = true
  容量 = 1.0 (100%)

可组批的工序:
  op1: batch_family = ["A", "X"], batch_size = 2 (占用50%), process_time = 100
  op2: batch_family = ["A123"], batch_size = 2 (占用50%), process_time = 120
  op3: batch_family = ["B"], batch_size = 1 (占用100%), process_time = 80
  op4: batch_family = ["A_type", "B_type"], batch_size = 4 (占用25%), process_time = 110
  op5: batch_family = ["C"], batch_size = 2 (占用50%), process_time = 90

合法批次1: {op1, op2}
  - 家族兼容: fuzzy_match("A", "A123") = True (子串匹配)
  - 容量: 0.5 + 0.5 = 1.0 ≤ 1.0 ✓
  - 批次时间: max(100, 120) = 120 分钟

合法批次2: {op1, op4}
  - 家族兼容: fuzzy_match("A", "A_type") = True (模糊匹配)
  - 容量: 0.5 + 0.25 = 0.75 ≤ 1.0 ✓
  - 批次时间: max(100, 110) = 110 分钟

合法批次3: {op4, op4_copy1, op4_copy2, op4_copy3}
  - 4个相同家族的工序
  - 容量: 0.25 × 4 = 1.0 ≤ 1.0 ✓

非法批次: {op1, op5}
  - 家族不兼容: batch_family[op1] = ["A", "X"]，batch_family[op5] = ["C"]
  - 所有字符串对都无法模糊匹配 ✗
```

### 3.12 特性12：大规模问题分解(Large-Scale Decomposition)

**定义**：对于大规模调度问题（1500+订单，50+设备），设计分解框架将问题分解为多个子问题求解。

**实现策略**（可选，后期实现）：

1. **时间窗分解**：
   - 将整个调度周期分解为多个时间窗
   - 每个时间窗内求解一个子问题

2. **工厂分解**：
   - 按工厂划分子问题
   - 处理跨厂转运的耦合关系

3. **产品家族分解**：
   - 按产品类型或家族分组
   - 减少批次组合的复杂度

4. **滚动窗口优化**：
   - 固定已调度的工序
   - 优化未来时间窗内的工序

**注意**：此特性可以在基础求解器完成后再实现。

## 4. 调度目标

### 4.1 主要目标

本问题采用多目标优化，目标按优先级排序如下：

#### 目标1（首要目标）：最大化全局截止时间内的总产出量

**定义**：在全局截止时间 `max_output_horizon` 内完成的订单的总产量尽可能大。

**数学表达式**：
```
Maximize: Σ (final_output_weight[i] × completed[i])

其中:
  completed[i] = 1  if completion_time[i] ≤ max_output_horizon
  completed[i] = 0  otherwise
```

**说明**：
- 不考虑每个订单的单独截止时间（`Task::delivery_deadline`）
- 只关注全局截止时间 `Instance::Config::max_output_horizon`
- 优先完成高产出重量的订单
- 这是最重要的优化目标

#### 目标2（次要目标）：最小化设备切换总次数

**定义**：在所有设备上进行切换（Setup）的总次数尽可能少。

**数学表达式**：
```
Minimize: Σ Σ setup_count[m, i, j]
          m  i,j

其中:
  setup_count[m, i, j] = 工序i和工序j在设备m上连续加工的次数
```

**说明**：
- 减少设备切换可以节省切换时间，提高设备利用率
- 尽量将相同或相似类型的工序连续安排在同一设备上
- 组批加工可以有效减少切换次数

#### 目标3（第三目标）：最大化订单按时达成率

**定义**：在各自截止时间前完成的订单比例尽可能高。

**数学表达式**：
```
Maximize: (1 / N) × Σ on_time[i]

其中:
  on_time[i] = 1  if completion_time[i] ≤ delivery_deadline[i]
  on_time[i] = 0  otherwise
  N = 订单总数
```

**等价形式**：
```
Minimize: tardy_count = Σ (1 - on_time[i])
```

**说明**：
- 考虑每个订单的单独截止时间 `Task::delivery_deadline`
- 按时完成率 = 按时完成的订单数 / 总订单数
- 这是第三优先级的目标，在满足前两个目标的前提下尽量提高

### 4.2 目标优先级说明

目标之间的优先级关系为**字典序优化**：

```
目标1 >> 目标2 >> 目标3

即:
1. 首先最大化全局截止时间内的总产出量
2. 在目标1相同的情况下，最小化设备切换次数
3. 在目标1和目标2都相同的情况下，最大化订单按时达成率
```

**优化策略**：
- 可以采用加权和方法：`F = w1×目标1 + w2×目标2 + w3×目标3`，其中 `w1 >> w2 >> w3`
- 也可以采用约束优化方法：先优化目标1，然后在保持目标1不变的前提下优化目标2，以此类推
- 对于大规模问题，建议使用启发式算法逐步改进解

### 4.3 次要考虑因素

在主要目标之外，还可以考虑以下因素（权重较低）：

- 最小化设备空闲时间
- 最小化跨厂转运次数
- 均衡设备负载
- 优先完成高优先级订单（`Task::priority`）

## 5. 输入输出规范

### 5.1 输入数据

输入数据采用JSON格式，包含以下字段：

1. **调度配置**：
   - `schedule_start_date`: 调度开始时间
   - `cal_threads`: 计算线程数
   - `max_output_horizon`: 最大产出时间范围

2. **工厂与设备信息**：
   - 工厂列表：工厂ID、名称
   - 设备列表：设备ID、名称、所属工厂、组批类型、维修时段

3. **任务信息**：
   - 任务ID、名称
   - 最早开始时间、交付截止时间
   - 优先级、产出重量
   - 加工路径列表

4. **工序信息**：
   - 工序ID、序号
   - 候选设备列表及加工时间
   - 组批属性
   - 时间约束（最小/最大间隔）

5. **时间矩阵**：
   - 设备切换时间矩阵
   - 跨厂转运时间矩阵

### 5.2 输出数据

输出数据采用JSON格式，包含以下字段：

1. **调度方案**：
   - 工序ID、任务ID、路径ID
   - 分配的设备ID
   - 开始时间、结束时间
   - 所属批次ID（如果组批）

2. **调度指标**：
   - 总完工时间(Makespan)
   - 加权总完工时间
   - 延迟订单数量和延迟时间
   - 设备利用率

3. **甘特图数据**（可选）：
   - 便于可视化的调度方案数据

## 6. 约束总结

### 6.1 硬约束（必须满足）

1. **工艺约束**：
   - 工序必须按照加工路径的顺序执行
   - 每个工序必须分配到一台候选设备上

2. **设备约束**：
   - 同一设备在同一时间只能加工一个工序（非组批）或一个批次（组批）
   - 设备在维修时段内不可用

3. **时间约束**：
   - 工件的第一道工序不能早于释放时间开始
   - 工序之间的最小间隔约束
   - 工序之间的最大间隔约束

4. **设备切换约束**：
   - 同一设备上的连续工序之间需要切换时间

5. **跨厂转运约束**：
   - 跨厂工序之间需要转运时间
   - 只能在允许的工厂对之间转运

6. **组批约束**：
   - 只有同一family的工序才能组批
   - 批次容量不能超过1

7. **路径选择约束**：
   - 每个任务只能选择一条加工路径

### 6.2 软约束（尽量满足）

1. **交付期约束**：
   - 尽量在截止日期前完成订单

2. **优先级约束**：
   - 优先完成高优先级任务

## 7. 求解器要求

### 7.1 算法要求

- 支持大规模问题求解（1500+订单，50+设备）
- 在合理时间内（如1小时）给出可行解
- 提供解的质量评估

### 7.2 性能要求

- 支持多线程并行计算
- 内存使用合理
- 可中断和恢复计算

### 7.3 可扩展性

- 易于添加新的约束
- 易于调整目标函数权重
- 支持参数配置
